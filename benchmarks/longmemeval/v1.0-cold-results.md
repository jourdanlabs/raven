# LongMemEval — RAVEN v1.0 Cold-Run Baseline

**Status:** Cold-run baseline. NO TUNING. Defaults only.
**RAVEN version:** 1.0.0 (commit `8a8f629`)
**Dataset:** `longmemeval_oracle.json` from
[`xiaowu0162/longmemeval-cleaned`](https://huggingface.co/datasets/xiaowu0162/longmemeval-cleaned)
(2025/09 cleaned release, 500 instances)
**Run date:** 2026-04-27
**Wall time:** 39.6 s for the full 500-question pass on a MacBook (M-series, in-process)

---

## Configuration

Pipeline construction (per question, fresh in-memory store):

```python
from raven.storage.store import RAVENStore
from raven.pipeline import RAVENPipeline

store = RAVENStore(":memory:")                    # default TFIDFEmbedder
pipeline = RAVENPipeline(store=store, top_k=20)   # everything else default
```

| Knob                  | Value                                |
| --------------------- | ------------------------------------ |
| `top_k`               | 20                                   |
| `aurora_threshold`    | `aurora.APPROVE_THRESHOLD` = 0.80    |
| `half_life_days`      | `eclipse.DEFAULT_HALF_LIFE_DAYS` = 30 |
| `embedder`            | `TFIDFEmbedder` (lexical, char-trigram, 512-dim) |
| `semantic embeddings` | **OFF** (no `sentence-transformers`) |
| `meteor_config`       | default `METEORConfig()` (DEFAULT_ALIASES only) |

Each LongMemEval `question` produces one fresh pipeline; each haystack turn is
ingested as one `MemoryEntry` (text = `[USER]/[ASSISTANT] <content>`,
timestamp parsed from session date, no entity_tags or topic_tags injected).


**Legend**

- `sR@k` — session-level recall at k: did RAVEN's top-k retrieved entries include any
  turn from an evidence session?
- `tR@k` — turn-level recall at k: did RAVEN's top-k include the specific
  `has_answer=True` turn?
- `A@k` — answer-substring hit rate: does any of the top-k entries contain the
  normalized gold-answer string (with ≥0.5 token-overlap fallback)?
- `absAcc` — abstention accuracy on the 30 `_abs` questions (REFUSED status OR no
  answer-substring hit in top-5).

Ranking source for top-k is `(approved + rejected)` sorted descending by
`ScoredMemory.retrieval_score`. RAVEN's AURORA approval gate does not enter the
ranking — see "What this number does NOT measure" below.

## Comparison to MemPalace's published 96.6%

MemPalace publishes a **96.6%** number on LongMemEval. RAVEN's cold-run **A@5
= 73.4%** is roughly **23 percentage points behind**. We are not going to spin
this. Two things are honest to say about the gap:

1. **Methodology asymmetry.** MemPalace's 96.6% is the official LongMemEval
   metric — a GPT-4o judge grading natural-language answers generated by a
   memory-augmented LLM. RAVEN v1.0 is a memory-validation pipeline that
   returns ranked memory entries, not synthesized answers. A direct apples-to-
   apples comparison would require routing RAVEN's top-k through an LLM
   answerer and re-running the official `evaluate_qa.py`. That's deferred as
   a Phase 2.1 follow-on (see GAPS additions below). The number reported here
   is **lexical answer-presence in top-k**, which is a *lower bound* on what a
   downstream LLM could synthesize correctly.

2. **The capabilities that close most of the gap are not yet shipped.** Phase
   2.1 introduces semantic embeddings, per-class decay, and answer-aware
   ranking heuristics that should materially improve the multi-session and
   temporal-reasoning categories — exactly the two categories where RAVEN is
   weakest in this cold-run. Cold-run is the bar we measure those
   improvements against.

The cold-run number is the **starting line**, not the verdict.

## Loss profile — top failure categories

100% of failures (125 / 470 answerable questions) are **Mode B**: RAVEN's
retrieval located the evidence session in top-5 (sR@5 > 0 for every failure),
but the gold answer text didn't substring-match any of the top-5 retrieved
turns. Failures by category:

| question_type             | failures / answerable | % |
| ------------------------- | --------------------: | ---: |
| single-session-preference |                23 / 30 | 76.7% |
| multi-session             |               51 / 121 | 42.1% |
| temporal-reasoning        |               39 / 127 | 30.7% |
| knowledge-update          |                 9 / 72 | 12.5% |
| single-session-user       |                 3 / 64 |  4.7% |

### Top 5 failure exemplars

**1. Single-session-preference** (76.7% miss rate)
- Q: *"Can you recommend some resources where I can learn more about video editing?"*
- Gold: *"The user would prefer responses that suggest resources specifically tailored to Adobe Premiere Pro…"*
- RAVEN: REFUSED. sR@5=1.00, tR@5=1.00 — perfect retrieval, but the gold field is a **rubric describing the desired response style**, not text expected to appear in memory. Substring-match is a poor metric here. This category penalizes RAVEN's retrieval quality unfairly.

**2. Multi-session counting** (e.g. `gpt4_59c863d7`)
- Q: *"How many model kits have I worked on or bought?"*
- Gold: *"five model kits"* (often gold is just the integer `5`).
- RAVEN: REFUSED. sR@5=1.00 — found all evidence sessions, but the answer requires **counting across sessions**, which RAVEN's retrieval pipeline does not perform. This is a synthesis failure, not a retrieval failure.

**3. Temporal-reasoning** (e.g. `gpt4_2655b836`)
- Q: *"What was the first issue I had with my new car after its first service?"*
- Gold: *"GPS system not functioning correctly"*
- RAVEN: REFUSED. sR@5=0.67, tR@5=0.50 — found 2/3 evidence sessions but missed the specific issue turn. Selecting "first" requires ordering by date which lexical TFIDF does not weight.

**4. Knowledge-update yes/no** (e.g. `d7c942c3`)
- Q: *"Is my mom using the same grocery list method as me?"*
- Gold: *"Yes."*
- RAVEN: REFUSED. sR@5=1.00, tR@5=1.00 — perfect retrieval. Gold is a one-word
  inference. Substring "yes" is not in evidence verbatim; the answer is implicit.
  Synthesis failure.

**5. Single-session-user paraphrase** (e.g. `75499fd8`)
- Q: *"What breed is my dog?"*
- Gold: *"Golden Retriever"*
- RAVEN: REFUSED. sR@5=1.00, tR@5=0.00 — retrieved evidence session, but the
  specific turn naming the breed wasn't in top-5. tR@5 was zero on this one,
  suggesting the breed-naming turn lost to other turns in the same session.

### Failure summary

- **Synthesis-required (counting / yes-no / paraphrase across sessions):** the
  dominant failure mode. Substring matching of gold answers requires that the
  gold text appear verbatim in some retrieved turn. For multi-session counting
  ("how many"), yes/no inference, and rubric-style preference golds, this is
  structurally impossible regardless of retrieval quality.
- **Turn-level ranking inside a recovered session:** session_recall@10 = 98.7%
  vs turn_recall@10 = 80.2%. RAVEN almost always retrieves the *right session*
  but sometimes loses the *right turn within it* to other turns from the same
  session that share more lexical overlap with the query.
- **AURORA over-refusal on chat-turn corpora:** 473 / 500 questions ended in
  REFUSED status (94.6%). Approved-memory count was zero on **all 500**
  questions. The 0.80 APPROVE_THRESHOLD plus 30-day decay applied to raw chat
  turns means no entry crossed the bar. The retrieval ranking itself is healthy
  — the AURORA gate is calibrated for fact-style memories (MUNINN-style) rather
  than conversational turns. This is a calibration finding, not a retrieval
  finding, and is in scope for Phase 2.1.

## What this number does NOT measure

1. **No LLM-generated answers.** Official LongMemEval requires answer
   *generation*; we report top-k *retrieval presence* of the gold answer
   substring. This is a strict lower bound on what an LLM-augmented
   downstream system would score.
2. **No semantic embeddings.** TFIDFEmbedder is character-trigram lexical only.
   Queries that paraphrase ("breed of dog" vs "Golden Retriever") suffer.
   `SentenceTransformerEmbedder` is opt-in and **not enabled** for this run.
3. **No counting / yes-no inference.** RAVEN does not synthesize answers, so
   counting questions ("how many X") and yes/no inference questions
   ("is X true?") are structurally penalized by substring scoring even when
   retrieval is perfect.
4. **No date-aware ordering.** Temporal-reasoning questions like "the first
   issue" require selecting evidence by date order. RAVEN's QUASAR+ECLIPSE
   weights are time-aware but not query-aware-of-temporal-intent.
5. **No per-question-type tuning.** All categories share the same TFIDF
   weights, decay rate, and AURORA threshold. Phase 2.1 will tune these.
6. **No corpus pre-processing.** Each haystack turn is ingested as a single
   memory entry without summarization, entity extraction, or session
   compression. A real RAVEN deployment would likely store distilled
   memories rather than raw turns.
7. **No abstention recall comparison.** The 100% abstention accuracy is
   inflated by RAVEN's aggressive REFUSED rate (any system that refuses every
   query gets 100% abstention recall). Combined with the ~94.6% refusal rate
   on answerable questions, the abstention number on its own is not a
   meaningful capability claim — it must be read alongside the refusal rate.
8. **METEORConfig contains custom JL-domain aliases.** The DEFAULT_ALIASES
   dictionary in `meteor.py` includes Jourdan Labs–specific entities (TJ,
   Bulma, COSMIC, MineralLogic, etc.). These do not overlap with LongMemEval
   content and are noise here, but they are still loaded.

## Reproducibility

```bash
# 1. Get the dataset (~15 MB)
mkdir -p /tmp/longmemeval_data && cd /tmp/longmemeval_data
curl -LO https://huggingface.co/datasets/xiaowu0162/longmemeval-cleaned/resolve/main/longmemeval_oracle.json

# 2. Run the cold-run harness from repo root
cd /path/to/raven
PYTHONHASHSEED=0 .venv/bin/python -m benchmarks.longmemeval.harness \
    --output v1.0-cold-results.json
```

Setting `PYTHONHASHSEED=0` makes the TFIDFEmbedder bucket hashing deterministic.
Without it, the per-question scores vary by ~1–2% but the per-type ordering
and overall numbers are stable. We checked `PYTHONHASHSEED=0` vs `=42`:
A@5 was 0.734 vs 0.749 — within 1.5% noise.
